## 1. Контекст

1. У проекта есть конкретный первый заказчик.
2. Владелец дерева живет в Европе, часть семьи живет в РФ.
3. Цель ближайшего этапа - не "универсальный genealogy SaaS", а production-ready версия для одной реальной семьи.
4. Эта версия должна быть достаточно надежной и удобной, чтобы владелец мог сам вести дерево, отправлять ссылки родственникам, приглашать помощников, собирать медиа от семьи и не зависеть от разработчика в ежедневной работе.
5. После этого опыта запуска будет собираться отдельная массовая версия продукта.

## 2. Продуктовая рамка версии

### 2.1 Название этапа

1. Рабочее название: `V1.0 Slava edition`.

### 2.2 Главная цель

1. Запустить приватную, стабильную и удобную версию для одной семьи.
2. Отработать на реальном сценарии:
- модель дерева,
- роли доступа,
- семейный шаринг по ссылке,
- медиа-поток,
- базовую админку,
- эксплуатацию и поддержку.
3. Не тратить время V1 на массовые сценарии, которые заказчику сейчас не нужны.

### 2.3 Ключевые принципы

1. `Owner-led collaboration`: главный пользователь - один владелец, но продукт должен нормально работать и для приглашенных родственников.
2. `Private-by-default, shareable-by-choice`: это семейный архив, который владелец может открывать по приглашениям и ссылкам.
3. `Low-ops`: минимум ручной поддержки и минимум инфраструктурной сложности.
4. `EU + RF ready`: дерево и медиа должны быть проверены в Европе и в РФ.
5. `Архитектура без овербилда`: делаем правильно там, где потом будет больно переделывать, но не строим enterprise-систему раньше времени.

## 3. Принципы доменной модели для V1

### 3.1 Что делаем правильно уже сейчас

1. Связь остается отдельной сущностью между двумя людьми, а не свойством карточки персоны.
2. Поддержка нескольких партнерств у одного человека закладывается уже в V1, даже если в семье заказчика это не основной сценарий.
3. История пары может хранить статус и даты, потому что это относится к конкретной связи, а не к человеку в целом.

### 3.2 Что сознательно НЕ делаем в V1

1. Не вводим обязательную публичную типизацию детей как `adopted` против `biological`.
2. В V1 действует принцип: ребенок - это ребенок, полноценный член семьи без отдельной иерархии "настоящий / не настоящий".
3. Для первой версии достаточно базовой модели `родитель -> ребенок` без этически спорной детализации.
4. Если позже в массовой версии потребуется хранить специальные семейные обстоятельства, это должно быть отдельным, аккуратно продуманным решением, а не дефолтной категоризацией ребенка.

### 3.3 Практический вывод для V1

1. Для детей и родителей в `Slava edition` достаточно базовой связи `parent-child`.
2. Для взрослых связей достаточно `partner/ex-partner` с возможностью нескольких партнерств у одной персоны.
3. Сложные типы `step/foster/guardian/adopted` в V1 не входят.

## 4. Что входит в V1.0 Slava edition

### 4.1 Управление деревом

1. Регистрация и вход владельца.
2. Одно дерево на владельца остается допустимым ограничением.
3. Создание, редактирование и удаление персон.
4. Назначение корневой персоны дерева.
5. Быстрое добавление родителя, ребенка и партнера с canvas.

### 4.2 Роли и доступ

1. Роли `owner`, `admin`, `viewer`.
2. Владелец может приглашать админов и зрителей.
3. Админ может редактировать дерево, загружать и привязывать медиа, но не владеть системой целиком.
4. Viewer может просматривать доступные данные через аккаунт.
5. В V1 также входит семейный шаринг по ссылке для read-only сценариев.
6. Ссылки должны быть управляемыми: создать, скопировать, отозвать.
7. Дерево по умолчанию приватное, любой доступ открывается только явным действием владельца.

### 4.3 Связи

1. Родитель-ребенок.
2. Партнерство.
3. Поддержка нескольких партнерств у одной персоны.
4. Возможность удалить или скорректировать связь.
5. Валидация против циклов и явных структурных ошибок.

### 4.4 Медиа

1. Единый поток загрузки для `photo`, `video`, `document`.
2. Приватная выдача по signed URL.
3. Отказ от обязательной модели `public video link`.
4. Привязка медиа к персоне.
5. Базовые metadata: имя, подпись, mime, размер, владелец, дата загрузки.
6. Для V1 видео считается обычным приватным файлом; сложный streaming/transcoding pipeline не нужен.
7. Приглашенные родственники с правом редактирования могут догружать свои файлы сами.

### 4.5 Рабочие экраны

1. Dashboard владельца.
2. Builder/viewer дерева.
3. Участники и приглашения.
4. Экран/блок share links для семейного read-only доступа.
5. Настройки дерева.
6. Журнал действий владельца.

### 4.6 Эксплуатационная готовность

1. Базовый backup и план восстановления.
2. Понятные ошибки при загрузке медиа и редактировании связей.
3. Минимальный monitoring/alerting для production.
4. Smoke-check сценариев владельца и администратора.
5. Проверка доступа из Европы и из РФ перед запуском.

## 5. Что НЕ входит в V1

1. Массовый onboarding для большого числа внешних клиентов.
2. Подписки, платежи, тарифы, биллинг.
3. GEDCOM import/export как пользовательская функция.
4. Merge дублей персон.
5. Публичные SEO-страницы, marketing-like story pages и открыто индексируемое распространение.
6. Field-level privacy policy-engine для публичного дерева.
7. Расширенная taxonomy родства (`step/foster/guardian/adopted`).
8. Терабайтные объемы, транскодинг, CDN-оптимизации "на вырост".

## 6. Технические решения для Slava edition

### 6.1 Что сохраняем из текущей базы проекта

1. `Next.js + TypeScript + Supabase` остается базовым runtime для первой версии.
2. Текущая модель `one tree per owner` подходит под задачу и не требует срочного расширения.
3. Роли `owner/admin/viewer` остаются основой permission model.
4. Для V1 допускается разделение access-модели на два канала:
- invite-based collaboration для тех, кто помогает редактировать,
- link-based read-only sharing для родственников, которым нужен только просмотр.

### 6.2 Что меняем/доводим в первую очередь

1. Медиа должны перейти к единой приватной модели вместо split-сценария `photo private + video public link`.
2. Relationship UX нужно упростить под реальные действия владельца: добавить родителя, ребенка, партнера, убрать лишнюю терминологию.
3. Owner-flow должен стать самодостаточным: без ручных вмешательств разработчика в обычной работе.
4. Family collaboration flow тоже должен стать самодостаточным:
- владелец отправляет ссылку родственнику,
- родственник открывает дерево в read-only режиме,
- выбранные помощники принимают приглашение и догружают медиа/правки без участия разработчика.

### 6.3 Рекомендация по media storage для V1

1. Для `Slava edition` важнее надежная кросс-региональная доступность и низкая стоимость, чем гипермасштаб.
2. Для текущего проекта `Cloudflare R2` rollout для новых upload является обязательной частью `Slava edition`, а не пост-launch улучшением.
3. `Yandex Object Storage` в текущем execution path остается transitional compatibility/read path для уже загруженных файлов до явного завершения migration.
4. Причины:
- в репозитории уже есть `R2` foundation, rollout metadata и direct-upload validation path,
- запуск не должен оставлять продукт в half-migrated состоянии,
- новая binary plane должна быть закрыта в рамках `Slava edition`, а не вынесена в "потом".
5. Архитектурное правило:
- приложение должно работать через storage adapter,
- vendor не должен быть жестко зашит в бизнес-логику,
- смена провайдера в V2 должна быть возможна без полной переделки media-domain.
6. `Cloudflare Stream`, `Queues` и self-managed `FFmpeg/HLS` не являются обязательной частью запуска, пока `R2/private delivery` закрывает реальные сценарии.

### 6.4 Приватность для V1

1. Основной механизм приватности V1 - приватное дерево плюс роли.
2. В V1 входит не публичный SEO-режим, а контролируемый семейный доступ по приглашениям и ссылкам.
3. Сложный field-level privacy engine переносится в более поздний этап.
4. Если нужны чувствительные заметки, их лучше сразу закладывать как `owner/admin only`, а не как публичные поля с будущими фильтрами.

## 7. Поэтапный план реализации

### 7.1 Этап A - Product freeze для Slava edition

1. Зафиксировать scope первой версии.
2. Явно вынести mass-market backlog за пределы V1.
3. Зафиксировать простую relationship policy:
- дети без subtype taxonomy,
- партнерства с поддержкой нескольких связей,
- private-by-default.
4. Зафиксировать access policy:
- invite-based collaboration для edit/upload,
- link-based sharing для read-only family access,
- revoke как обязательная часть V1.

**Результат этапа**
1. Команда и заказчик одинаково понимают, что входит в запуск, а что нет.

### 7.2 Этап B - Owner/Family collaboration flows

1. Довести CRUD персон до состояния "владелец справляется без разработчика".
2. Довести быстрые действия добавления связей с канваса.
3. Проверить редактирование и удаление связей.
4. Довести приглашения и роли.
5. Сделать семейный шаринг по ссылке отдельным, понятным и управляемым сценарием.
6. Привести dashboard и builder к спокойной, понятной иерархии без лишнего visual noise.

**Acceptance criteria**
1. Владелец может сам собрать и править дерево.
2. Можно безопасно делегировать часть работы одному-двум администраторам.
3. Владелец может отправить read-only ссылку родственнику без ручной помощи разработчика.
4. Ошибки в типовых сценариях редки и понятны пользователю.

### 7.3 Этап C - Unified private media

1. Убрать зависимость от публичных видео-ссылок как основного сценария.
2. Сделать один upload/access flow для фото, видео и документов.
3. Привязать медиа к людям и проверить права доступа.
4. Ограничить размеры файлов и поддерживаемые типы для V1.
5. Продумать удаление файлов и cleanup orphaned objects.

**Acceptance criteria**
1. Владелец и админ могут загрузить медиа без обращения к разработчику.
2. Родственник с правом редактирования может сам догрузить свои файлы.
3. Viewer или получатель share-link видит только разрешенные файлы.
4. Медиа открываются и в Европе, и в РФ.

### 7.4 Этап D - Mandatory Cloudflare rollout

1. Проверить gated-конфигурацию `Cloudflare R2`:
- `CF_R2_*`,
- bucket CORS,
- upload-intent metadata,
- `smoke:media`,
- `smoke:media:direct`.
2. Активировать rollout через `CF_R2_ROLLOUT_AT`.
3. Подтвердить, что новые upload идут в `cloudflare_r2`.
4. Подтвердить, что legacy Yandex-backed медиа продолжают читаться.
5. Зафиксировать rollout как обязательную часть launch checklist.

**Acceptance criteria**
1. Новые upload больше не зависят от legacy Yandex path.
2. Legacy media остается читаемой в transition.
3. Rollout state наблюдаем через код, smoke и operational notes, а не через догадки по env.

### 7.5 Этап E - Production hardening

1. Настроить production env, backup и базовые операционные процедуры.
2. Пройти smoke/regression для ключевых маршрутов.
3. Проверить приглашения, share links, права, аудит, загрузку и просмотр медиа.
4. Провести реальное UAT с владельцем, хотя бы одним помощником и хотя бы одним read-only родственником из РФ.
5. Подготовить короткий owner guide: как приглашать, редактировать дерево, добавлять медиа, что делать при ошибках.

**Acceptance criteria**
1. Есть рабочий checklist запуска.
2. Есть способ восстановить данные после сбоя.
3. Есть подтвержденный сценарий доступа из обеих географий.

## 8. Ориентир по срокам

1. Так как у проекта уже есть рабочая база, `Slava edition` логично планировать как `2-4 недели` концентрированной доводки, а не как greenfield-разработку.
2. Грубая разбивка:
- `3-5 дней` на product freeze и уточнение scope,
- `1-2 недели` на core flows,
- `1-2 недели` на unified media и hardening,
- `несколько дней` на UAT, правки и launch checklist.
3. Если всплывет инфраструктурная проблема доступа из РФ, срок надо считать отдельно, потому что это уже не UI-polish, а platform-level задача.

## 9. Критерии готовности V1

1. Владелец сам создает и поддерживает дерево.
2. Владелец сам приглашает помощников и зрителей.
3. Владелец сам отправляет read-only ссылки родственникам.
4. Можно добавлять людей, связи и медиа без участия разработчика.
5. Медиа приватны и не опираются на публичные consumer-links.
6. Новые upload проходят через `Cloudflare R2`, а legacy Yandex-backed медиа остаются читаемыми в transition.
7. Приложение открывается и работает для владельца в Европе и для родственников в РФ.
8. Есть backup/restore дисциплина.
9. Продукт решает задачу одной семьи без явных "заглушек для будущего SaaS".

## 10. Что переносится в V2 mass-market

1. GEDCOM import/export.
2. Merge duplicates.
3. Story pages, viral/public sharing и SEO-сценарии.
4. Более глубокая privacy-модель.
5. Расширенная taxonomy родства, если она вообще будет нужна после реального пользовательского опыта.
6. Массовый onboarding, подписки, тарификация и growth-механики.
7. Более сложная media-инфраструктура.

## 11. Главный вывод

1. Первая версия должна быть не "маленьким черновиком будущего SaaS", а законченным private product для одной семьи.
2. Архитектурно уже сейчас стоит правильно сделать:
- хранение связей как отдельных сущностей,
- поддержку нескольких партнерств,
- приватную media-модель,
- owner/admin/viewer permission model,
- контролируемый family sharing по приглашениям и ссылкам,
- обязательный `Cloudflare R2` rollout для нового upload path в рамках `Slava edition`.
3. Все остальное, что не помогает конкретному владельцу вести семейное дерево в production сегодня, должно быть отложено до V2.
